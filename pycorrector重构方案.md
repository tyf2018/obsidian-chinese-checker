# 1. 结论

当前插件的主要问题已经从“pycorrector 能否被调用”转向“误判率偏高，结果可信度不足”。现阶段如果继续把主要精力放在历史性的启动链路细节上，会偏离当前真实矛盾。

1. 协议层曾出现 CORS/preflight 失败，导致服务在线但前端不可调用。
2. 启动与门控状态收敛不稳定，出现长时间 pending 与误回退。
3. 错误分类不一致，临时网络抖动被渲染为引擎异常。
4. 多引擎结果合并、规则置信度和去重策略设计偏弱，导致误判较多。
5. 在 fallback 模式下，文档纠错效果明显下降，用户感知为“检查不出来”。

修订后的目标顺序应为：

1. 先做误判止血，恢复结果可信度。
2. 再做性能与稳定性收尾，避免质量问题被“启动链路问题”掩盖。
3. 最后再做模块化与协议契约化。

# 3. 设计缺陷分析

## 3.1 协议层缺陷

1. 未把桌面端跨域约束视为必测项。
2. `/health` 与 `/check` 错误语义缺乏稳定契约，前端依赖字符串猜测。

## 3.2 生命周期缺陷

1. 探活、拉起、门控收敛没有形成严格闭环。
2. 启动阶段缺少明确“终态保证”（ready/unavailable/failed）。

## 3.3 状态机缺陷

1. `loading/init/unavailable/ready` 与 `pycorrector_available` 映射历史上存在偏差。
2. 错误字段有残留污染风险（旧错误覆盖新状态）。

## 3.4 质量层缺陷

1. fallback 与 pycorrector 的结果质量差异没有显式告知。
2. 可检测范围策略（frontmatter、代码块、链接、公式）缺少回归保障，易漏检。
3. JS 词典候选将“未登录词/OOV”近似视为“疑似错词”，对专有名词、领域术语、缩写词不友好。
4. 混淆词规则采用固定 `WORD_HINTS + CONFUSION_CHAR_MAP` 全量滑窗，一字之差即判错，语境约束不足。
5. pycorrector 结果同时走 `details + diff` 两条路径，可能产生“整词 + 子词片段”双命中，噪音偏大。
6. `mergeMatches()` 仅按精确 span 合并，不做“包含关系去重/跨引擎主次裁决”，会保留碎片化建议。
7. 当前全局阈值偏低（用户配置已到 `0.45`），而规则型结果的初始 confidence 又偏高，误判容易穿透过滤。
8. JS 与 Python 共用一个阈值口径，但两类 confidence 并不在同一统计尺度上，实际不可直接比较。

## 3.5 可观测性缺陷

1. 缺少请求级 Trace ID 与分段耗时。
2. 面板虽有回退原因，但仍不足以快速定位失败阶段。

## 3.6 首次安装与启用流程缺陷

1. 当前设置页虽然已提供 `.venv` 路径、安装命令、自检按钮，但仍更像“分散工具集合”，不是完整的首次安装流程。
2. “插件本体启用成功”和“Python 引擎可用”语义没有彻底拆开，用户容易把 Python 未就绪误判为插件加载失败。
3. 配置保存与配置验证分层不清，容易出现“路径已保存，但不可执行/不可写/依赖缺失”的半配置状态。
4. 缺少统一的首次安装状态机，至少应明确区分 `unconfigured`、`configured_unverified`、`ready`、`error`。
5. 自动启动前提不够保守，Python 未通过预校验时仍可能触发探活、拉起或误导性文案。
6. `install_pycorrector.bat` 仍带有较强入口耦合，但它本质上只能是 Windows 下的可选安装助手，不应成为安装闭环的核心假设。
7. 首次安装失败后的安全默认值不够明确；正确策略应是“插件先稳定启用为 JS 模式，再引导用户补齐 Python 能力”。

## 3.7 Python 运行时约束缺陷

1. 文档仍把 Python 解释器视为“可配置项”，但对当前插件而言，`pycorrector + kenlm + Windows` 的稳定组合实际上非常窄。
2. 现有方案没有把“仅支持 Python 3.11”写成硬约束，导致用户可能继续尝试 3.10/3.12/3.13，并把生态兼容问题误判成插件缺陷。
3. 首次安装流程虽强调自检，但没有把“不符合版本即直接阻断 Python 链路”写成明确策略，仍存在半可用、半失败状态。
4. `kenlm` 安装困难的根因之一就是运行时版本漂移；若不冻结 Python 版本，后续任何安装体验优化都会被解释器差异稀释。

# 4. 改造目标（修正后口径）

1. 手动触发准确率：误判样本集中，误报数较当前基线下降 >= 50%。
2. 手动触发可用性：手动触发时 `/check` 成功响应率 >= 95%（剔除用户主动中断）。
3. 引擎真实性：在手动触发且 `pycorrector_status=ready` 条件下，结果能区分“pycorrector 主导”和“规则补充”。
4. 状态一致性：不再出现“health=ready 但面板长期显示启动中/不可用”。
5. 可诊断性：任一失败在 1 分钟内可定位到协议、进程、门控或范围抽取阶段。
6. 自动触发约束：非手动场景不得调用 Python，也不得产生“Python 启动中”噪音提示。
7. 首次安装稳健性：即使 `.venv` 路径错误、目录不可写、依赖缺失或 Python 未安装，插件也必须稳定启用并退化到 JS 安全模式。
8. 安装交互清晰度：用户必须能明确知道自己处于“未配置 / 已配置未验证 / 可用 / 异常”哪一种状态，而不是靠报错文案自行推断。
9. 运行时收敛：插件的 Python 能力只在 `Python 3.11.x` 下被视为受支持，其余版本一律视为“未通过环境约束”，只允许退化到 JS。

# 6. 指标与口径（修正）

1. 手动触发成功率：手动触发下 `/check` 成功响应占比（剔除用户主动中断）。
2. pycorrector 主导率：手动触发下，结果中 `engine_detail` 含 `pycorrector` 的占比。
3. 超时率：`python_check_timeout` 在手动触发中的占比。
4. 范围抽取正确率：应检文本命中 + 不应检文本过滤准确率。
5. 文档命中率：回归文档中预期错误被命中的比例。
6. 误判率：误判样本中被错误标记的问题数 / 样本总数。
7. 片段重复率：同一错误位点出现“整词 + 子词片段”双提示的比例。
8. OOV 误报占比：CEDICT OOV 规则引起的误报占全部误报的比例。
9. 非手动调用率：非手动场景进入 Python 检测链路的占比，目标为 0。
10. 首次安装安全启用率：空白环境下插件能成功启用并进入 JS 安全模式的占比，目标为 100%。
11. Python 配置完成率：用户按设置页引导完成 `.venv`、依赖安装与自检后达到 `ready` 的占比。
12. Python 版本合规率：进入 `ready` 的环境中，`Python 3.11.x` 占比应为 100%。

# 11. 新增需求（2026-03-09）

## 11.0 首次安装环境参数与依赖交互稳健性评估

现状判断：
1. 当前设置页已经具备 `.venv` 路径、Python 可执行路径、安装命令预览、安装助手、自检按钮等基础能力。
2. 这些能力仍偏“离散操作项”，尚未形成稳定的首次安装闭环。
3. 当前最大风险不是某个 bat 或命令失败，而是 Python 未准备好时，用户把“能力未就绪”误判成“插件整体无法启用”。

欠妥之处：
1. 安装入口仍偏工程师视角，默认假设用户理解 `.venv`、解释器、依赖、自检之间的先后关系。
2. 设置保存后若不立即校验，容易留下半配置状态，后续启动阶段才暴露问题。
3. `pythonAutoStart` 若不受 `ready` 状态约束，会把首次安装期的临时错误放大为加载期噪音。
4. `install_pycorrector.bat` 当前仍带有较强存在感，但它只能覆盖部分 Windows 场景，无法承担跨环境稳定入口职责。
5. 方案没有把运行时版本冻结写成硬约束，仍给用户留下“任意 Python 版本都可尝试”的错误预期。

稳健化原则：
1. 插件启用成功优先于 Python 可用。
2. JS 安全模式优先于激进自动启动。
3. 配置校验通过优先于“保存即生效”。
4. 向导式步骤优先于散点式按钮集合。
5. 运行时冻结优先于“尽量兼容更多 Python 版本”；当前只支持 `Python 3.11.x`。

建议交互顺序：
1. 首次安装时默认展示“当前处于 JS 安全模式，Python 能力尚未配置”。
2. 先校验 Python 版本；只有 `Python 3.11.x` 才允许进入后续步骤，其余版本直接提示不受支持。
3. `.venv` 路径默认值继续使用 `S:\obsidian-chinese-checker\.venv`，但用户可修改；保存前先做存在性与可写性预校验。
4. 只有通过预校验后，才显示“安装依赖”作为下一步主动作。
5. 安装完成后必须显式执行一次自检，自检通过才允许切换到 Python/混合模式。
6. 任一步失败都回落到明确状态和下一步提示，不得让插件进入“看似已启用，实际持续报错”的悬空状态。

关于 `install_pycorrector.bat` 的结论：
1. 有保留价值：对 Windows 用户仍可作为一键安装助手，减少手工输入命令成本。
2. 不应承担核心职责：它不应决定插件能否启用，也不应是唯一安装路径。
3. 正确定位应为“可选助手 + 命令预览入口”，而不是“安装闭环核心组件”。

关于 Python 版本的明确规定：
1. 当前插件的 Python 能力仅支持 `Python 3.11.x`。
2. `3.11.x` 之外的版本，即使能手动安装部分依赖，也不视为受支持环境。
3. 自检、设置页、安装脚本、诊断文案必须统一使用同一口径，避免“能跑但不受支持”的灰色状态。

# 12. 误判问题专项分析（2026-03-09）

## 12.1 当前误判的主要根因

结合当前实现，误判主要来自以下 6 类：

1. `CEDICT OOV 候选过强`
   - 代码位置：`main.js` 中 `detectCedictCandidates()`
   - 当前逻辑会对 2-6 字中文 token 做“词典缺失 -> 近形/同音替换候选”。
   - 问题在于“词典里没有”不等于“写错了”，对专有名词、组织名、地名、新词尤其容易误判。

2. `混淆提示规则语境过弱`
   - 代码位置：`main.js`、`python_engine_service.py` 中 `WORD_HINTS / CONFUSION_CHAR_MAP`
   - 当前是一字差异 + 固定候选词的滑窗匹配，没有句法、词性、领域词过滤。
   - 这类规则适合做补充，不适合作为高置信主结果。

3. `pycorrector 结果被二次扩展成碎片`
   - 代码位置：`python_engine_service.py` 中 `_detect_by_pycorrector()`
   - 现在同时解析 `details` 和 `diff`，容易把一个整词错误拆成“整词命中 + 单字命中”两条结果。
   - 这会显著放大用户对误判和噪音的感知。

4. `合并逻辑只按精确 span，不按语义主次`
   - 代码位置：`main.js` 中 `mergeMatches()`，`python_engine_service.py` 中 `_merge_match_groups()`
   - 目前不会主动淘汰“被更长 span 包含的低价值结果”，因此碎片建议会保留下来。

5. `置信度口径不统一`
   - 当前 pycorrector、Python fallback、JS hint、CEDICT OOV 的 confidence 都在 0-1 区间，但来源不同。
   - 现在却共用统一阈值过滤，且用户当前阈值已调低到 `0.45`，会放大规则型误判。

6. `缺少领域词白名单`
   - 目前 `filterMatches()` 只看全局阈值、用户词典、会话忽略。
   - 缺少 vault 级术语表、插件名/命令名、常见技术词、中英混排保护。

## 12.2 改进思路

1. 先做“减噪”，再做“提召回”
   - 当前首先要解决的是“用户不信任结果”，不是继续堆更多候选。

2. 把“词典型候选”降级为低置信来源
   - CEDICT OOV 不能直接当错别字结论，应该视为“候选提示”。

3. 以 pycorrector 为主、规则为辅
   - pycorrector 负责真正的上下文纠错。
   - 规则只补 pycorrector 的漏检，不应与 pycorrector 平权展示。

4. 结果展示分层
   - 高置信建议默认展示。
   - 中低置信建议折叠，或至少在 UI 上明确标注“候选提示”。

5. 去掉碎片化建议
   - 对同一起点附近的“整词/子词”冲突，保留整词、高优先级结果。

6. 建立误判回归样本
   - 没有误判样本集，就无法证明“误判下降”。

## 12.3 hint（混淆提示）的改进空间

现状判断：
1. `hint` 不是第三方现成能力，本质上是当前插件自定义的“混淆字/混淆词提示规则”。
2. 当前问题不在“有没有规则”，而在“触发面过宽、语境约束过弱、展示权重过高”。

主要缺陷：
1. `滑窗范围过宽`
   - 任何满足一字差异的短语都可能触发，容易污染技术文档、参数说明、命令示例。
2. `缺少词边界意识`
   - 现在更多是字符窗口命中，而不是先识别词边界再判断，容易把局部片段误当成自然语言词语。
3. `缺少支持信号`
   - 只要命中混淆映射就可能提示，没有要求 pycorrector、固定短语、词典候选等二次确认。
4. `展示层级过高`
   - 作为“提示型规则”，当前仍可能与 pycorrector 结果在同一层级展示，用户会误以为这是强结论。

改进方向：
1. 从“全量滑窗”改为“词边界感知”
   - 只在中文连续词片段内运行，不对代码味/路径味/参数味较强的行启用。
2. 增加语境约束
   - 要求候选词与上下文长度、位置、词典状态相匹配；必要时引入左右邻词黑名单。
3. 增加支持信号
   - 仅当满足以下至少一项时提升为正式结果：
   - 与高精度固定短语规则一致
   - 与 pycorrector 在相近 span 上同向
   - 命中高频词且原词不在白名单
4. 调整展示定位
   - `hint` 默认应视为“低置信候选提示”，而非“直接纠错结论”。
5. 单独回归
   - 为技术术语、命令示例、路径说明、UI 文案建立 hint 误判专项回归。

预期收益：
1. 明显降低 `hint` 在技术文档中的误判。
2. 保留其对常见单字混淆的补漏价值。
3. 避免用户把提示型规则误解为模型结论。

## 12.4 fallback（Python 规则）的改进空间

现状判断：
1. `fallback` 同样不是第三方现成规则库，而是当前插件 Python 服务里的自定义兜底规则。
2. 它适合承接“高精度固定错误”，不适合继续承担“泛化纠错”职责。

主要缺陷：
1. `规则职责边界不清`
   - fallback 既承担兜底，又在某些场景下与 pycorrector 平权展示，定位偏乱。
2. `全局扫描成本高`
   - 当前与 pycorrector 同请求执行时，可能对全文再次扫描，增加耗时。
3. `规则精度分层不足`
   - 固定短语、重复字、机会主义替换规则若混在一起，会拉低整体可信度。
4. `规则源未完全单一`
   - Python fallback 与 JS 规则仍有重复维护风险，后续容易漂移。

改进方向：
1. 收缩职责
   - fallback 只负责“高精度确定性规则”，例如固定短语、重复字、明确错别字对。
2. 改成局部补漏
   - 优先只在 pycorrector 未覆盖或未命中的局部范围运行，而不是全文平权扫描。
3. 规则分层
   - 将 fallback 规则分为：
   - `核心高精度规则`：默认启用，可直接展示
   - `扩展候选规则`：默认关闭或降级展示
4. 单一规则源
   - 与 JS 侧共享规则文件、字段结构与测试基线，避免双端语义漂移。
5. 强制回归
   - 每新增一条 fallback 规则，都必须附对应正例/反例回归。

预期收益：
1. 提升 fallback 的可信度，让其真正承担“兜底”而非“放大噪音”。
2. 降低 Python 侧额外扫描带来的时间成本。
3. 降低 JS/Python 双端规则漂移导致的维护成本。

## 12.5 实施优先级建议

1. 第一批：`P1-6-C/D/B`
   - 先做包含关系去重、pycorrector 结果融合、分引擎阈值。
   - 这是最快能明显降低误判体感的改动。

2. 第二批：`P1-6-F/E`
   - 再做 CEDICT OOV 收紧、领域白名单。

3. 第三批：`P1-6-I/J`
   - 再对 hint/fallback 做职责收缩、局部补漏和规则分层。

4. 第四批：`P1-6-G/H`
   - 最后做 UI 分级展示和误判回归集固化。
