# 1. 结论

当前插件的主要问题不是 pycorrector 本身，而是“调用链路稳定性”和“状态表达一致性”。

1. 协议层曾出现 CORS/preflight 失败，导致服务在线但前端不可调用。
2. 启动与门控状态收敛不稳定，出现长时间 pending 与误回退。
3. 错误分类不一致，临时网络抖动被渲染为引擎异常。
4. 在 fallback 模式下，文档纠错效果明显下降，用户感知为“检查不出来”。

目标是先把“可调用性和可诊断性”做稳定，再提升文档场景纠错效果。

# 3. 设计缺陷分析

## 3.1 协议层缺陷

1. 未把桌面端跨域约束视为必测项。
2. `/health` 与 `/check` 错误语义缺乏稳定契约，前端依赖字符串猜测。

## 3.2 生命周期缺陷

1. 探活、拉起、门控收敛没有形成严格闭环。
2. 启动阶段缺少明确“终态保证”（ready/unavailable/failed）。

## 3.3 状态机缺陷

1. `loading/init/unavailable/ready` 与 `pycorrector_available` 映射历史上存在偏差。
2. 错误字段有残留污染风险（旧错误覆盖新状态）。

## 3.4 质量层缺陷

1. fallback 与 pycorrector 的结果质量差异没有显式告知。
2. 可检测范围策略（frontmatter、代码块、链接、公式）缺少回归保障，易漏检。

## 3.5 可观测性缺陷

1. 缺少请求级 Trace ID 与分段耗时。
2. 面板虽有回退原因，但仍不足以快速定位失败阶段。

# 4. 改造目标（修正后口径）

1. 自动触发稳定性：自动触发下 `engine_source=js` 占比 >= 99%，且不出现 Python 回退噪音。
2. 手动触发可用性：手动触发时 `/check` 成功响应率 >= 95%（剔除用户主动中断）。
3. 引擎真实性：在手动触发且 `pycorrector_status=ready` 条件下，`engine_source` 含 `python/混合` 占比 >= 95%。
4. 状态一致性：不再出现“health=ready 但面板长期显示启动中/不可用”。
5. 可诊断性：任一失败在 1 分钟内可定位到协议、进程、门控或范围抽取阶段。

# 11. 新增需求（2026-03-09）

## 11.1 JS 引擎增强：评估 `cedict_ts.u8` 可行性

输入文件：`E:\OneDrive\1黑曜石【工作】\.obsidian\plugins\various-complements\cedict_ts.u8`

现状确认：
1. 文件可读，约 9.2MB，`entries=122970`，格式为 CC-CEDICT（`繁体 简体 [拼音] /释义/`）。
2. 当前 JS 引擎仍是“小规则集”：固定错词、重复字、极小混淆集，覆盖面有限。

结论：
1. 可用于增强 JS，引擎能力会明显好于当前固定规则。
2. 仅靠 CEDICT 不能直接替代上下文纠错模型；它更适合做“词典增强 + 候选生成 + 误报抑制”。
3. 不能在运行时每次全量解析原始 `.u8`；必须预编译索引后再加载。

风险与约束：
1. 路径依赖风险：直接依赖 `various-complements` 插件目录不稳，目标插件禁用或迁移会失效。
2. 许可证风险：CC-BY-SA 4.0 需要保留署名与共享条款，发布包需补齐 `NOTICE/LICENSE`。
3. 质量边界：缺少词频与上下文概率时，建议只把 CEDICT 结果作为“中低置信候选”，避免误报上升。

执行清单：已并入第 5 章 `P1-5`，本节仅保留背景分析，避免任务重复维护。

## 11.2 pycorrector 耗时过久：根因与改进思路

已定位根因（结合现有代码路径）：
1. 服务端 `detect_with_meta()` 对整段文本调用 `_PYCORRECTOR_CORRECT_FN(text)`，即使前端传了 `ranges`，pycorrector 仍处理全文。
2. 同一次请求内还会继续跑 hint/fallback 全文扫描，导致长文总耗时叠加。
3. 前端超时后主动中断连接，服务端写响应时出现 `WinError 10053`；这是“超时后果”，不是首因。

执行清单：已并入第 5 章 `P0-5`，本节仅保留根因分析，避免任务重复维护。
