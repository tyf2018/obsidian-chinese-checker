# 1. 结论

当前插件的主要问题已经从“pycorrector 能否被调用”转向“误判率偏高，结果可信度不足”。现阶段如果继续把主要精力放在历史性的启动链路细节上，会偏离当前真实矛盾。

1. 协议层曾出现 CORS/preflight 失败，导致服务在线但前端不可调用。
2. 启动与门控状态收敛不稳定，出现长时间 pending 与误回退。
3. 错误分类不一致，临时网络抖动被渲染为引擎异常。
4. 多引擎结果合并、规则置信度和去重策略设计偏弱，导致误判较多。
5. 在 fallback 模式下，文档纠错效果明显下降，用户感知为“检查不出来”。

修订后的目标顺序应为：

1. 先做误判止血，恢复结果可信度。
2. 再做性能与稳定性收尾，避免质量问题被“启动链路问题”掩盖。
3. 最后再做模块化与协议契约化。

# 3. 设计缺陷分析

## 3.1 协议层缺陷

1. 未把桌面端跨域约束视为必测项。
2. `/health` 与 `/check` 错误语义缺乏稳定契约，前端依赖字符串猜测。

## 3.2 生命周期缺陷

1. 探活、拉起、门控收敛与 `ready` 后补跑闭环虽已在代码中补齐，但仍缺少端到端回归数据验证。
2. 启动阶段已有 `ready/unavailable/failed` 终态，但门控与手动窗口、补跑逻辑耦合较深，后续维护成本偏高。

## 3.3 状态机缺陷

1. `loading/init/unavailable/ready` 与 `pycorrector_available` 映射历史上存在偏差。
2. 错误字段有残留污染风险（旧错误覆盖新状态）。

## 3.4 质量层缺陷

1. fallback 与 pycorrector 的结果质量差异没有显式告知。
2. 可检测范围策略（frontmatter、代码块、链接、公式）缺少回归保障，易漏检。
3. JS 词典候选将“未登录词/OOV”近似视为“疑似错词”，对专有名词、领域术语、缩写词不友好。
4. 混淆词规则采用固定 `WORD_HINTS + CONFUSION_CHAR_MAP` 全量滑窗，一字之差即判错，语境约束不足。
5. pycorrector 结果同时走 `details + diff` 两条路径，可能产生“整词 + 子词片段”双命中，噪音偏大。
6. 虽已补 `collapseContainedMatches()` 与服务端 `_collapse_contained_matches()`，但跨引擎主次裁决仍不够强，局部碎片建议仍可能漏出。
7. 已改为“全局阈值 + rule/source 阈值上取最大值”，但高歧义规则仍可能在技术文档中穿透过滤。
8. JS 与 Python 的 confidence 口径仍不统一，只是通过分引擎阈值做了第一层止血，尚未形成真正可比的评分体系。

## 3.5 可观测性缺陷

1. 缺少请求级 Trace ID 与分段耗时。
2. 面板虽有回退原因，但仍不足以快速定位失败阶段。

## 3.6 首次安装与启用流程缺陷

1. 设置页已具备步骤化提示、状态机文案、自检入口与安装命令预览，但默认配置仍不是“纯 JS 模式”，安全默认值仍依赖 guard 而非配置本身。
2. “插件本体启用成功”和“Python 引擎可用”的语义已大体拆开，但首装失败场景仍需专项回归验证。
3. 配置保存与配置验证已分层，但目录可写性、依赖缺失等更细粒度预校验仍未完全覆盖。
4. 首次安装状态机 `unconfigured / configured_unverified / ready / error` 已落地，后续重点转为验收和补齐剩余预校验。
5. 自动启动前提已收紧到 `ready + 最近自检通过 + Python 3.11.x`，但相关文案与回归数据仍需固化。
6. `install_pycorrector.bat` 的定位已弱化为可选助手，但设置页与文档仍需持续压低其“唯一入口”心智。
7. 首次安装失败后的运行时安全性主要靠 JS 回退与预校验兜底，尚未通过空白环境/错误路径专项回归证明。

## 3.7 Python 运行时约束缺陷

1. Python 解释器虽然仍可配置，但代码层已经把受支持运行时收敛到 `Python 3.11.x`，其余版本不会进入 `ready`。
2. 当前剩余问题不在“是否有 3.11 硬约束”，而在于这套约束还缺少跨场景回归与更细粒度失败归因。
3. 首次安装流程已经把“非 3.11 阻断 Python 链路”写进设置页、自检与启动逻辑，但空白环境默认值仍有进一步保守化空间。
4. `kenlm` 安装困难仍受运行时与本机构建链影响，因此版本冻结是必要前提，但不是安装体验问题的全部解法。

# 4. 改造目标（修正后口径）

1. 手动触发准确率：误判样本集中，误报数较当前基线下降 >= 50%。
2. 手动触发可用性：手动触发时 `/check` 成功响应率 >= 95%（剔除用户主动中断）。
3. 引擎真实性：在手动触发且 `pycorrector_status=ready` 条件下，结果能区分“pycorrector 主导”和“规则补充”。
4. 状态一致性：不再出现“health=ready 但面板长期显示启动中/不可用”。
5. 可诊断性：任一失败在 1 分钟内可定位到协议、进程、门控或范围抽取阶段。
6. 自动触发约束：非手动场景不得调用 Python，也不得产生“Python 启动中”噪音提示。
7. 首次安装稳健性：即使 `.venv` 路径错误、目录不可写、依赖缺失或 Python 未安装，插件也必须稳定启用并退化到 JS 安全模式。
8. 安装交互清晰度：用户必须能明确知道自己处于“未配置 / 已配置未验证 / 可用 / 异常”哪一种状态，而不是靠报错文案自行推断。
9. 运行时收敛：插件的 Python 能力只在 `Python 3.11.x` 下被视为受支持，其余版本一律视为“未通过环境约束”，只允许退化到 JS。

# 6. 指标与口径（修正）

1. 手动触发成功率：手动触发下 `/check` 成功响应占比（剔除用户主动中断）。
2. pycorrector 主导率：手动触发下，结果中 `engine_detail` 含 `pycorrector` 的占比。
3. 超时率：`python_check_timeout` 在手动触发中的占比。
4. 范围抽取正确率：应检文本命中 + 不应检文本过滤准确率。
5. 文档命中率：回归文档中预期错误被命中的比例。
6. 误判率：误判样本中被错误标记的问题数 / 样本总数。
7. 片段重复率：同一错误位点出现“整词 + 子词片段”双提示的比例。
8. OOV 误报占比：CEDICT OOV 规则引起的误报占全部误报的比例。
9. 非手动调用率：非手动场景进入 Python 检测链路的占比，目标为 0。
10. 首次安装安全启用率：空白环境下插件能成功启用并进入 JS 安全模式的占比，目标为 100%。
11. Python 配置完成率：用户按设置页引导完成 `.venv`、依赖安装与自检后达到 `ready` 的占比。
12. Python 版本合规率：进入 `ready` 的环境中，`Python 3.11.x` 占比应为 100%。

## 10.3 误判治理约束

1. 误判治理新增需求只允许落到 `P1-6-A` 到 `P1-6-J`。
2. 新增误判样例时，必须同时给出：
   - 预期建议。
   - 当前错误建议。
   - 应归属的任务项编号。
   - 验收口径。
3. 对 `己/已/几`、`的/地/得`、`在/再` 这类高歧义单字混淆，默认按 `P1-6-I/J` 处理，不得在没有语境约束和回归样例的情况下直接扩规则。
4. CEDICT/OOV 相关需求默认归 `P1-5` 或 `P1-6-F`，不得把 OOV 直接提升为高置信正式结论。
5. 专名、公文术语、平台名、机构简称类误判，默认优先归 `P1-6-E/H/J`，不得仅通过继续加单字混淆规则处理。
6. 同一原词在单文档内出现多次且误报候选不一致时，默认视为“一致性失效”问题，优先用文档级保护和结果裁决修复。
7. 人名、地名、门派名、器物名等文学专名误判，默认优先归 `P1-6-E/H/I/J`，不得仅按“常见字更常见”原则直接放行建议。
8. 对白/引号/章节叙述中的误判，默认先做文体降权和专名回判，不得直接扩更多通用错字规则。

## 10.4 协议与工具约束

1. 协议字段或诊断格式变更，必须同步更新：
   - `docs/protocol.md`
   - `tools/contract_check.py`
   - 第 5 章中受影响任务的验收口径
2. 范围抽取相关需求，必须同步更新 `tools/range_extractor_check.js` 或 `docs/regression_cases.md` 中对应样例。
3. 发布门禁新增项，必须直接写回第 9 章，不再在修订记录或附录中保留临时清单。
