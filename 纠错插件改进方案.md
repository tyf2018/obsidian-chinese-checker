# 纠错插件改进方案

## 1. 当前问题归纳

1. 启动阶段提示噪音：Obsidian 刚启动时 Python 服务尚未就绪，容易出现 `Failed_to_fetch`，用户看到“引擎不可用”提示但稍后其实可恢复。  
2. 回退原因可读性不足：技术原因（网络、模型、依赖、启动中）在 UI 上混淆为同一类失败。  
3. 引擎模式心智成本偏高：用户难以理解“仅 Python/混合/仅 JS”在失败场景下的行为差异。  
4. 诊断链路断点：有自检能力，但缺少标准化诊断报告与一键收集。  
5. 维护复杂度上升：`main.js` 逻辑集中，未来迭代风险高。

## 2. 目标

1. 启动阶段“少打扰”：把短暂不可达视为“启动中”而非“不可用”。  
2. 失败原因“可定位”：同类错误统一映射，给出明确修复路径。  
3. 模式语义“可预期”：模式定义与实际行为一致。  
4. 诊断“可复现”：用户和开发者看到一致的状态证据。  
5. 代码“可演进”：逐步拆分，降低回归成本。

## 3. 已落地改进（本轮）

1. 启动健康门控（P0-1）：首次检测前最多等待约 10 秒健康就绪，未就绪按“启动中”处理。  
2. 启动期容错：`Failed_to_fetch` 在自动拉起场景下转为 `python_booting`，避免无意义报错。  
3. 启动超时语义优化：进程存活但未就绪时不再直接判死，后台继续探活。  
4. 模式文案汉化：`Hybrid` -> `混合（推荐）`。  
5. 命令合并：由“启动/停止”改为“重载引擎”。  
6. 依赖与模型修复：`kenlm` 已可用，语言模型已落地，`pycorrector_status` 可达 `ready`。  

## 4. 后续改进空间（按优先级）

### P0（稳定性）

1. 启动健康门控：
   - 首次检测前先做短周期 `health` 轮询（最多 8-12 秒）。
   - 未就绪仅显示“启动中”，不显示“失败”。
2. 错误分类统一表：
   - 统一 `python_unreachable / python_booting / python_unavailable / python_error` 的映射规则。
   - 结果面板显示“建议动作”（例如：重载引擎、运行自检）。
3. 端口冲突探测：
   - 启动前检测端口占用并提示 PID/进程名（仅诊断展示，不强杀）。

### P1（准确率与体验）

4. 模式策略可配置：
   - 增加“Python 严格模式下空结果是否允许回退 JS”的开关（默认关闭）。
   - 混合模式增加“重叠候选保留策略”选项（保守/激进）。
5. 结果解释增强：
   - 每条建议增加来源标签（pycorrector/规则/混淆提示）与置信度。
   - 支持“按来源过滤”与“只看高置信度”。
6. 自检增强：
   - 在自检结果中输出：Python 路径、模型路径、端口、引擎状态、最近错误。
   - 提供一键复制“问题报告文本”用于反馈。

### P2（工程可维护性）

7. 前端模块拆分：
   - `engine_client`、`engine_manager`、`range_extractor`、`result_view`、`settings_view`。
8. 协议测试常态化：
   - 把 `tools/contract_check.py` 纳入发布前检查。
9. 回归样本集：
   - 建立 100-200 条常见错词/行业术语样本，跟踪误报率与漏报率。

## 5. 验收指标

1. 启动后 15 秒内，“引擎不可用”提示次数下降 90% 以上。  
2. 典型 5k 字文档实时检测，P95 响应时间 < 2.5 秒。  
3. 误报反馈闭环可用（忽略/词典后同文本不再重复弹出）。  
4. 自检输出能直接定位到“路径、模型、网络、协议、端口”之一。

## 6. 建议执行顺序

1. 先做 P0-1/2（启动门控 + 错误统一映射）。  
2. 再做 P1-4/5（模式策略 + 结果解释）。  
3. 最后做 P2（模块拆分 + 回归体系）。
