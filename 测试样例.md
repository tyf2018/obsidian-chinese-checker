# pycorrector 测试样例（v2）

目标：验证“仅手动检查时使用 Python”策略是否生效，并确认 pycorrector 在手动场景可用。

## 1. 预检查

1. 打开插件设置，确认：
2. `engineMode=hybrid` 或 `python`
3. `pythonEngineEnabled=true`
4. `pythonAutoStart=true`
5. `pythonManualTriggerOnly=true`
6. `pythonExecutable=S:\obsidian-chinese-checker\.venv\Scripts\python.exe`
7. 在设置页点击一次“引擎自检”。
8. 预期：能看到服务状态，不出现长期 `pending`。
9. `jsCedictEnhanced=true`
10. CEDICT 状态显示“已加载 N 词”（若未加载，执行设置页“复制构建命令”后在终端运行，再点“重载索引”）。

## 2. 自动触发验证（应只走 JS）

1. 打开任意 Markdown 文件，触发以下任一场景：
2. `live`（输入文本后等待自动检测）
3. `active-leaf-change`（切换文件）
4. `file-open`（打开文件）
5. 打开结果面板触发 `panel`
6. 在面板复制诊断 JSON，检查字段：
7. `engine_source` 应为 `js`
8. `fallback_reason` 应为空字符串

验收：自动触发不调用 Python，不出现 `python_booting/python_check_timeout/python_unreachable` 提示噪音。

## 3. 手动触发验证（应允许调用 Python）

1. 在面板点击“检查当前文件”，或执行命令“中文纠错：检查当前文件”。
2. 使用以下样例文本：

```text
今天天齐不太好。配眼睛。
这个方案因该可以上线，他己经提交代码。
我们要在接再厉，按步就班推进。
```

3. 复制诊断 JSON，检查字段：
4. `trigger` 为 `manual` 或 `panel-button`
5. `engine_source` 优先包含 `python` 或 `混合`
6. 若失败，`fallback_reason` 需给出具体原因（例如 `python_unavailable:python_check_timeout`）
7. 若成功且本次使用 Python，`fallback_reason` 应为空

验收：手动触发能进入 Python 链路；失败时原因可解释，不再只显示笼统“启动中”。

## 4. 范围过滤验证

将以下内容保存为测试文档并手动触发检查：

~~~markdown
---
title: 前言天齐（frontmatter，不应检）
---

正文：今天天齐不太好。（应检）

`行内代码天齐`（不应检）

```js
const a = "代码块天齐"; // 不应检
```

公式：$天齐$（不应检）

链接：https://example.com/天齐/path（不应检）
~~~

验收：仅正文“天齐”参与检测，其余区域不应产生命中。

## 5. 长文稳定性验证（手动触发）

1. 准备 3000-8000 字中文长文（含代码块、链接、公式）。
2. 连续手动触发 3 次“检查当前文件”。
3. 每次记录：
4. `engine_source`
5. `fallback_reason`
6. `stage_durations.detectMs`
7. `python_last_error`
8. `python_last_stderr`

验收：
1. 自动触发路径无 Python 超时噪音。
2. 手动触发若超时，需明确显示 `python_check_timeout` 等具体原因。
3. 面板可复制完整诊断 JSON。

## 7. JS CEDICT 增强验证（自动触发）

1. 保持 `pythonManualTriggerOnly=true`，触发方式选 `live`。
2. 使用文本：

```text
今天的天齐不错，我们继续再接再励。
这个问题他己经处理，流程会按步就班进行。
```

3. 复制诊断 JSON 检查：
4. `engine_source=js`
5. `js_cedict_enabled=true`
6. `js_cedict_ready=true`

验收：在自动触发（纯 JS）场景，较改造前更容易命中“天齐/再接再励/己经/按步就班”等典型错误。
