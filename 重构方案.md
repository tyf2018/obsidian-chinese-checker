# 0. 目标与范围

## 0.1 目标
1. 让 Obsidian 中文纠错“稳定可用”，先解决不可用/回退问题，再提升准确率与性能。  
2. 在不破坏现有用户配置的前提下，允许代码重构。  
3. 给出可验收、可回滚、可分阶段交付的改造路径。

## 0.2 非目标
1. 本轮不引入云端依赖，不强制接入远端 LLM。  
2. 本轮不做“全自动替换”，保持“建议+人工确认”交互。  
3. 本轮不强制迁移到 TypeScript（可作为后续增强）。

---

# 1. 基线现状（以当前仓库为准）

## 1.1 已存在能力
1. 已有完整 JS 检测流程、Python 本地引擎、混合策略、设置页、结果面板、逐条应用/忽略/加入词典。  
2. 已有 `/health` 与 `/check` 协议、超时控制、自动拉起、故障熔断。  
3. 已有 Markdown 过滤（frontmatter、代码块、行内代码、公式、URL 等）。

## 1.2 当前主要问题
1. 环境可用性是首要阻断：`data.json` 的 Python 路径可能失效（例如不存在 `.venv`）。  
2. pycorrector 不可用时会长期回退 fallback，实际纠错能力显著下降。  
3. JS 与 Python fallback 规则存在漂移风险（词表并非单一数据源）。  
4. 实时检测当前是“全文送检”，长文性能与稳定性仍有优化空间。  
5. `main.js` 体量较大，维护成本高，可读性和可测试性不足。

---

# 2. 重构原则

1. 先修可用性，再做准确率与性能。  
2. 保持协议兼容优先：先兼容旧字段，再渐进升级。  
3. 每个阶段都要可单独发布、可回滚。  
4. 以数据验收，不靠体感：误报率、召回、延迟、超时率。  
5. 重构不等于重写：优先抽离与替换高耦合模块，不一次性推翻。

# 3. 目标架构（分层）

## 3.1 前端（Obsidian）
1. `DetectionCoordinator`：调度（实时/手动/全库扫描）。  
2. `RangeExtractor`：文本过滤与可检测区间计算。  
3. `EngineClient`：HTTP 协议、超时、重试、熔断。  
4. `ResultMerger`：多引擎合并、去重、排序。  
5. `SuggestionUI`：高亮、悬浮建议、结果面板操作。

## 3.2 后端（Python 本地服务）
1. `EngineAdapter`：pycorrector 解析与错误分类。  
2. `FallbackRuleEngine`：词表与混淆规则。  
3. `ResponseAssembler`：统一输出结构。  
4. `HealthReporter`：可用性与加载状态元信息。

## 3.3 共享资源
1. 统一规则数据源（JSON）：短语错词、混淆字、提示词。
2. 协议契约文档（请求/响应字段、兼容策略、版本规则）。
3. 回归测试样本（常见误报/漏报场景，目标 50+ 场景）。

## 3.4 依赖关系图

```
M0 (可用性止血)
  ↓
M0.5 (日志与监控) ← 为后续阶段提供可观测性基础
  ↓
M1 (协议契约) ← 必须先稳定协议
  ↓
M2 (规则统一) ← 依赖协议稳定性
  ↓
M3 (增量检测) ← 依赖规则统一 + 日志监控
  ↓
M4 (准确率优化) ← 依赖前面所有基础设施
```

## M3（P1，3 天）：检测链路重构（性能与稳定性）

### 目标
减少全文重复检测，提升长文体验。

### 改动

#### 1. 段落级增量检测

**段落边界定义**：
- 以双换行符 `\n\n` 或标题行（`# ` 开头）为段落分隔
- 单个段落最大 1000 字，超过则按句号分割

**上下文窗口**：
- 变更段落前后各 1 个段落（共 3 个段落）
- 避免跨段落错误遗漏（如”上一段末尾+本段开头”的错误）

**触发策略**：
- 编辑后 500ms 防抖
- 仅检测光标所在段落 + 上下文
- 全文检测保留为手动命令

**实现伪代码**：
```javascript
function onEdit(editor) {
  const cursorLine = editor.getCursor().line;
  const paragraphs = splitIntoParagraphs(editor.getValue());
  const targetIndex = findParagraphIndex(paragraphs, cursorLine);
```

#### 2. 内存缓存策略

**缓存键设计**：
```javascript
const cacheKey = `${filePath}:${segmentHash}:${engineMode}`;
```

**segmentHash 计算**：
```javascript
function computeSegmentHash(text) {
  // 使用 FNV-1a 哈希算法（快速、冲突率低）
  let hash = 2166136261;
  for (let i = 0; i < text.length; i++) {
    hash ^= text.charCodeAt(i);
    hash *= 16777619;
  }
  return hash >>> 0; // 转为无符号 32 位整数
}
```

**缓存失效策略**：
- LRU 淘汰（最大 100 个条目）
- 单条目最大 10KB
- 文件关闭时清除该文件的所有缓存
- 引擎模式切换时清除所有缓存

**缓存命中率监控**：
```javascript
const cacheStats = {
  hits: 0,
  misses: 0,
  get hitRate() { return this.hits / (this.hits + this.misses); }
};
```

#### 3. 优化合并策略

**当前问题**：重叠候选直接丢弃，可能丢失高质量建议。

**新策略**：
```javascript
function mergeOverlappingMatches(matches) {
  // 1. 按 offset 排序
  matches.sort((a, b) => a.offset - b.offset);

  // 2. 检测重叠
  const merged = [];
  for (const match of matches) {
    const overlapping = merged.filter(m =>
      m.offset < match.offset + match.length &&
      match.offset < m.offset + m.length
    );

    if (overlapping.length === 0) {
      merged.push(match);
    } else {
      // 3. 按置信度和来源排序
      const candidates = [...overlapping, match];
      candidates.sort((a, b) => {
        // 优先级：pycorrector > js_rule > confusion_map
        const sourceWeight = { pycorrector: 3, js_rule: 2, confusion_map: 1 };
        const scoreA = (a.confidence ?? 0.5) * sourceWeight[a.source];
        const scoreB = (b.confidence ?? 0.5) * sourceWeight[b.source];
        return scoreB - scoreA;
      });

      // 4. 保留 top-2（允许用户选择）
      merged.push(...candidates.slice(0, 2));
    }
  }

  return merged;
}
```

#### 4. 请求量控制

**同文件串行队列**：
```javascript
const fileQueues = new Map(); // filePath -> Promise

async function queueCheck(filePath, text) {
  if (!fileQueues.has(filePath)) {
    fileQueues.set(filePath, Promise.resolve());
  }

  const queue = fileQueues.get(filePath);
  const newQueue = queue.then(() => performCheck(text));
  fileQueues.set(filePath, newQueue);

  return newQueue;
}
```

**跨文件并发上限**：
```javascript
const concurrencyLimit = 3; // 可配置
const activeTasks = new Set();

async function checkWithLimit(filePath, text) {
  while (activeTasks.size >= concurrencyLimit) {
    await Promise.race(activeTasks);
  }

  const task = queueCheck(filePath, text);
  activeTasks.add(task);
  task.finally(() => activeTasks.delete(task));

  return task;
}
```

新词典与规则包均可单独禁用，不影响主流程。

# 5. 代码重构边界（允许重构但可控）

## 5.1 推荐拆分（不引入新构建链）

### 拆分原则
1. **单一职责**：每个模块只做一件事
2. **依赖倒置**：核心逻辑不依赖具体实现
3. **接口隔离**：模块间通过明确接口通信

### 具体拆分方案

将 `main.js` 内部逻辑按模块拆为：

```
src/
├── engines/
│   ├── js_rule_engine.js        # JS 规则引擎
│   ├── python_local_engine.js   # Python 本地引擎客户端
│   └── engine_interface.js      # 引擎接口定义
├── core/
│   ├── engine_manager.js        # 引擎管理与切换
│   ├── range_extractor.js       # 文本过滤与区间计算
│   ├── result_merger.js         # 结果合并与去重
│   └── cache_manager.js         # 缓存管理
├── ui/
│   ├── result_panel_view.js     # 结果面板
│   ├── tooltip_actions.js       # 悬浮提示
│   └── status_bar.js            # 状态栏指示器
├── utils/
│   ├── logger.js                # 日志工具
│   ├── diagnostics.js           # 诊断报告生成
│   └── config_loader.js         # 配置加载
└── main.js                      # 插件生命周期与命令注册
```

### 拆分顺序（渐进式）

**第 1 步：抽离纯函数**（无副作用，易测试）
- `range_extractor.js`：文本过滤逻辑
- `result_merger.js`：合并算法
- `utils/logger.js`：日志格式化

**第 2 步：抽离无状态服务**（不依赖全局状态）
- `engines/python_local_engine.js`：HTTP 客户端
- `utils/diagnostics.js`：诊断报告生成

**第 3 步：重构有状态管理**（最复杂，最后做）
- `core/engine_manager.js`：引擎状态机
- `core/cache_manager.js`：缓存状态

### 每次拆分后的验证清单
- [ ] 运行完整测试套件
- [ ] 确认功能无回退（手动测试核心场景）
- [ ] 提交独立 commit（便于回滚）
- [ ] 更新模块依赖图

## 5.2 暂不做
1. 暂不切换到 FastAPI/Flask（当前 HTTPServer 可用，先解决协议与可用性问题）。
2. 暂不引入数据库。
3. 暂不引入远端服务依赖。
4. 暂不引入构建工具（Webpack/Rollup），保持简单。

# 附录：关键指标定义

## A.1 性能指标

- **平均检测耗时**：所有检测请求的平均响应时间
- **P95 检测耗时**：95% 的请求在此时间内完成
- **P99 检测耗时**：99% 的请求在此时间内完成
- **超时率**：超过配置超时时间的请求占比
- **回退率**：因 pycorrector 不可用而回退到 fallback 的请求占比
- **缓存命中率**：从缓存直接返回结果的请求占比

## A.2 准确率指标

- **误报率**：标记为错误但实际正确的占比
- **召回率**：实际错误中被检测出的占比
- **F1 分数**：精确率和召回率的调和平均数

## A.3 可用性指标

- **引擎可用率**：pycorrector 正常工作的时间占比
- **服务可用率**：后端服务正常响应的时间占比
- **MTBF**（平均故障间隔时间）：两次故障之间的平均时间
- **MTTR**（平均修复时间）：从故障发生到恢复的平均时间
